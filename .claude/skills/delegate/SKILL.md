# 멀티모델 위임 파이프라인

당신(리드)은 직접 구현하지 않는다. 분해, 라우팅, 중재, 통합만 한다.

## 1. 분해

요청을 독립적으로 실행 가능한 서브태스크로 나눈다. 각 서브태스크에 대해 명시한다:
- 목표와 완료 기준 (검증 가능한 형태로)
- 수정 허용 범위 (파일/디렉토리)
- 다른 서브태스크와의 의존 관계

## 2. 라우팅

| 작업 성격 | 담당 |
|---|---|
| 아키텍처 판단, 복잡한 리팩터링, 까다로운 디버깅, 미묘한 로직 | `opus-worker` |
| 보일러플레이트, 단순 CRUD, 대량 반복 수정, 포맷 변환, 빠른 프로토타입 | `codex-worker` |
| 코드베이스 탐색, 사전 조사 | `Explore` |
| 판단이 애매하면 | `opus-worker` (안전한 쪽) |

같은 파일을 두 워커가 동시에 수정하게 될 경우 Agent 호출에 `isolation: "worktree"`를 쓰거나, 순차 실행으로 바꾼다.

## 3. 병렬 실행

의존 관계 없는 서브태스크는 한 메시지에서 동시에 Agent 호출한다. 각 워커에게는 서브태스크의 목표·완료 기준·수정 범위를 프롬프트에 그대로 넣어준다 — 워커는 이 대화를 볼 수 없다.

## 4. 교차검증 (핵심 규칙)

**구현한 모델 계열과 다른 계열이 리뷰한다:**
- `codex-worker`가 구현 → `opus-reviewer`가 리뷰
- `opus-worker`가 구현 → `codex-worker`에게 리뷰 지시 (read-only 모드)
- codex 사용 불가 환경이면 → `opus-reviewer`로 대체하되, 구현 에이전트와 별도 인스턴스로 새로 띄운다

리뷰어에게는 변경된 파일 목록과 원래 요구사항을 전달한다.

## 5. 중재

리뷰 판정이 FAIL 또는 PASS_WITH_FIXES면:
1. 리드가 지적 사항을 코드를 직접 읽어 사실 확인한다 (리뷰어도 틀릴 수 있다)
2. 타당한 지적만 골라 원래 구현 담당 워커에게 수정을 재위임한다 (SendMessage로 같은 에이전트를 이어서 쓰면 컨텍스트가 유지됨)
3. 수정 후 재검증. 2회 왕복 후에도 평행선이면 리드가 직접 판정하고 사유를 기록한다

## 6. 통합·보고

- 전체 diff를 리드가 최종 확인 (범위 밖 수정, 워커 간 충돌 여부)
- 가능하면 실제 실행/테스트로 최종 검증
- 사용자 보고: 무엇이 바뀌었는지, 누가 무엇을 했는지, 교차검증에서 걸러진 것, 남은 위험
